# Test Feature

## Goal

Catch bugs before they reach review:
- Type checker passes clean (no implicit `any`, no type errors)
- Linter and formatter are green
- UI renders in browser without runtime errors
- Happy path + critical edge cases behave as specified in acceptance criteria

## Input

Context file path — usually `.tmp/<task-slug>/notes/context.md`. Read it to find:
- Acceptance criteria (what to test)
- `## Changed files` (current source of truth for feature surface)
- `## Iteration log` (if you bailed out on a previous iteration, check which bugs were supposedly fixed — verify they're actually gone)

## Steps

### 1. Static checks first (fast, cheap)

Run the project's type checker and linter (see PROJECT.md for the exact commands):

```bash
<type-check-command> 2>&1 | tee .tmp/<task-slug>/notes/ts-check.log
<lint-command>       2>&1 | tee .tmp/<task-slug>/notes/lint.log
```

If either fails:
- Parse the error list
- Write `.tmp/<task-slug>/test-specs/bugs.md` with each error as a "bug"
- Return `status: fail` — orchestrator loops back to Dev phase

Do NOT try to fix lint errors yourself. Report only. That's the contract with `implement-feature`.

### 2. Decide if UI verification is needed

| Touched files include | UI verification? |
|-----------------------|------------------|
| Component files (`.tsx`, `.jsx`, `.vue`, etc.) | YES |
| Style files (`.css`, `.scss`, `.module.css`, etc.) | YES |
| Router config | YES |
| Pure utility files (`.ts`, `.js`) | NO |
| Types / interfaces only | NO |
| API layer (only contract changes) | NO |

If NO → static checks already passed → return `status: pass`.

If YES → continue to step 3.

### 3. Generate test cases via the `test-writer` agent

Test cases are no longer authored inline in this skill. They are produced by
the `test-writer` sub-agent, which owns a persistent memory file of past
project bugs and automatically emits regression cases when the current change
surface can reach them.

#### 3.1. Build a clean test-context file

Write `.tmp/<task-slug>/notes/test-context.md` with:
- `## Goal` — one-sentence feature goal (copy from parent context)
- `## Acceptance criteria` — the checklist from parent context
- `## Changed files` — full current content of each changed file (clean slice,
  no iteration log, no previous bug reports, no implementor reasoning)

Mirror the shape of `review-context.md` from Phase 5 — it's the same pattern.

#### 3.2. Dispatch `test-writer` in generate mode

```
Agent({
  subagent_type: "test-writer",
  description: "Generate test cases with memory",
  prompt: "Generate atomic Playwright-ready test cases for the feature described in .tmp/<task-slug>/notes/test-context.md. Follow your instructions: read your memory, read the context, cross-check memory against the change surface, emit regression cases for any reachable past bug. Return the result as the structured Markdown block described in your output format. Do not write to ai.md — I will save your response."
})
```

The agent will return a Markdown block with cases labeled `G<n>` for generated
categories and `R<n>` for regression cases pulled from memory.

#### 3.3. Save the agent's response to `ai.md`

Write the agent's returned block verbatim to
`.tmp/<task-slug>/test-specs/ai.md`, prefixed with a single header line:

```markdown
# Test cases — generated by test-writer agent on <ISO timestamp>
```

Do not reformat or edit the agent's output — its format is already the
contract this skill consumes in step 4.

**Sanity check:** if the agent returned zero cases or reported that
`test-context.md` was empty, that's an orchestrator bug — stop and fix the
context file, do NOT paper over it with inline cases.

#### 3.4. When to call `/remember-edge-case` (context for the main Claude)

This skill does not call `/remember-edge-case`. That skill is invoked **by
the main Claude** after the user reports a bug during manual QA and you've
fixed it. The entry lands in `test-writer.memory.md`, and the next run of
step 3.2 picks it up as a regression case automatically.

### 4. Run component preview verification

Invoke the existing `test-on-sandbox` skill via the `Skill` tool. It handles:
- Dev server spin-up on a free port
- Component registration in the preview environment (adds if missing)
- Playwright MCP navigation to the component preview
- Screenshot capture to `.tmp/<task-slug>/screenshots/`
- Teardown of dev server + browser

For each test case from `ai.md`:
1. Navigate to the preview entry
2. Perform the action
3. Screenshot: `.tmp/<task-slug>/screenshots/case-<N>-<short-name>.png`
4. Verify the expected outcome against the screenshot and page snapshot
5. Check browser console for uncaught errors (fail the case if any)

### 5. Collect bug report

For every failing case write to `.tmp/<task-slug>/test-specs/bugs.md`:

```markdown
# Bug Report — iteration <N>

## Bug 1: <short title>
- **Test case**: Case 3 from ai.md
- **Expected**: empty-state placeholder with "No data"
- **Actual**: white screen, console error `Cannot read properties of undefined`
- **Screenshot**: .tmp/<task-slug>/screenshots/case-3-empty-state.png
- **Severity**: critical | high | medium | low
- **Likely location**: src/features/foo/ui/Foo.tsx line N
- **non_obvious**: true | false
```

Severity rubric:
- **critical** — crashes, data loss, broken core flow
- **high** — broken edge case that's in acceptance criteria
- **medium** — visual regressions, UX bugs not in criteria
- **low** — cosmetic, nice-to-have

**`non_obvious` rubric.** This field exists so the orchestrator can route
non-obvious failures through the `debug-hypothesis` skill instead of
bulldozer-retrying `implement-feature`. You are classifying *obviousness of
the root cause*, not severity. Classify each bug, then stop — the routing
decision belongs to `feature-workflow`, not to you.

- `non_obvious: true` — blank screen with unexplained stack trace;
  intermittent timing failure; data corruption under a race; state that
  contradicts the reducer; any failure whose direct cause is not visible in
  the last 30 lines of the current diff. These route through
  `debug-hypothesis` before the next Dev iteration.
- `non_obvious: false` — lint error, type error, missing import, typo in a
  label, trivial off-by-one in a loop visible in the diff, missing `id`
  prop on a form control, wrong text. These loop back to
  `implement-feature` directly. No hypothesis phase needed.

When in doubt, pick `true`. A wrongly-flagged trivial bug costs one extra
skill dispatch; a wrongly-flagged non-obvious bug costs a whole wasted Dev
attempt on a guess.

### 6. Exit decision

- All tests pass → return `status: pass` to orchestrator
- Any bugs → return `status: fail` with path to `bugs.md`

The return contract includes the per-bug `non_obvious` field inside
`bugs.md`. The orchestrator reads that field and decides whether to
dispatch `debug-hypothesis` before re-entering Phase 3 Implement. **Do not
pre-classify bugs by severity to decide routing yourself** — severity and
obviousness are orthogonal. A cosmetic `low` bug can still be non-obvious
(e.g. a layout collapse that happens only under a specific viewport width
because of a CSS containment rule in a parent module), and a critical crash
can be trivially obvious (e.g. a null deref visible in the first line of
the diff).

## Output

- `.tmp/<task-slug>/test-specs/ai.md` — test cases (persist across iterations, append new ones if feature grows)
- `.tmp/<task-slug>/test-specs/bugs.md` — overwritten each iteration (only current failures)
- `.tmp/<task-slug>/screenshots/case-*.png` — evidence
- `.tmp/<task-slug>/notes/{ts-check,lint}.log` — static check output
- Summary to caller: `status: pass | fail`, counts, paths

## Constraints

- **Screenshots only under `.tmp/<task-slug>/screenshots/`** — never in the project root or other locations
- Do NOT attempt to fix bugs yourself — you're the tester, not the implementor

## Learnings

- Step 3 delegates case authoring to the `test-writer` agent, which owns a
  persistent memory file (`.claude/agents/test-writer.memory.md`). Past
  project bugs turn into automatic regression coverage when their conditions
  are reachable on the current change surface.
- If `test-writer` returns zero cases or reports an empty context file, that
  is an orchestrator bug in step 3.1 — fix the context file, do NOT paper
  over it with inline cases.
- To teach the agent a new edge case after manual QA, the main Claude invokes
  the `/remember-edge-case` skill after fixing the bug. That skill dispatches
  `test-writer` in memorize mode; this skill then picks the entry up on the
  next run automatically.
